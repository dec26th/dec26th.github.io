  • TCP三次握手，TCP四次挥手

    • 为什么三，不是二？

      两次可能会出现建立多个链接的情况。例如发送方因为各种原因没有收到接收方的确认报文，则会再次向接收方发送connection请求，但是这时上一次发送的确认报文又被发送方收获，同时接收方相应第二次的连接请求，再次建立一个Tcp Connection。而三次握手则可以很好的避免该情况发生。

    • 四次握手的具体内涵

      • 第一次：客户端通知服务器关闭链接，发送包，其中FIN = 1，seq = x.

      • 第二次：服务器给客户端发送确认包，表示收到请求，但是并没有准备好关闭链接 ACK = 1，ACKnum = x + 1.

      • 第三次：服务器给客户端发送连接关闭请求，FIN = 1,seq = y.

      • 第四次：客户端收到服务端关闭连接响应，发送确认包。其中ACK = 1，ACKnum = y + 1服务器收到确认后，关闭连接。客户端在等待固定时间没有收到重传报文后，自己也关闭连接。

  • TCP可靠性

    • ARQ

      自动重传请求

    • 流量控制

      • 防止接收方从缓冲区读取数据速度慢
      • 控制发送方发送速率，防止接收方缓存溢出

    • 拥塞控制

      • 非正式定义：过多源发送过多数据，超过网络的处理能力

      • 不用于流量控制：会产生丢包(路由器缓冲区溢出)，延时长(路由器缓冲区排队)

      • TCP拥塞控制算法：

        • 慢启动：

          建立连接时，ConWin = 1 MSS。而可用宽带 >>> 1 MSS 以指数的速度增加发送速率，直到发生一个丢包发生为止。

        • 乘性减：

          发生丢包时间以后，ConWin 大小减半，门限值设置为当前ConWin的一半。

        • 加性增：

          窗口增大到门限值以后，线性增大。

    • 校验和

      TCP报文拥有校验和的字段，用于校验报文端是否出错

    • HTTPS协议是SSL + HTTP协议构建的可进行加密传输，身份认证的网络协议，相较于HTTP协议更加的安全。

  • Cookie和Session的区别
    • Cookie保存在客户端而Session保存在服务器端。
    • Cookie相对比较不安全，可以被分析然后用来Cookie诈骗。而Session相对来说更加的安全可靠。
    • Session由于存放在服务器端，当访问多时，影响服务器的性能。
    • 单个Cookie的大小限制是3K。

  • HTTP1.0和HTTP1.1和HTTP2.0的区别

    • HTTP1.0与HTTP1.1之间的区别

      • 长连接

        1.0默认短链接，1.1默认的长连接

      • 节约带宽

        1.0请求资源时，只能请求资源的所有，而对于1.1，他能够请求资源的部分

      • 缓存处理

      • 错误状态响应码不同

      • HOST域

        1.0默认一台主机只有一个IP地址，不需要添加hostname字段，而随着虚拟机技术的发展，一台物理机就可以对应了多台虚拟机，可以通过hostname指定虚拟机，若不指定虚拟机则会报错。

    • HTTP1.1与HTTP2.0的区别

      • 多路复用

        1.x采用keep alive 字段来解决复用TCP的问题，但是仍然存在着请求阻塞的问题。一条连接只能对应一个请求。而对于2.0，TCP链接可以并发的发起多个请求，且每个请求以stream的方式传输，每个stream有唯一的标识。

      • 数据压缩

        2.0支持header的压缩，而1.1不支持

      • 服务器推送

        服务器会顺便把一些客户端需要的资源一起推送给客户端，免得客户端再次创建链接发送请求到服务器端获取，方便获取各种静态资源。

    • 参数传递

      • POST用于创建资源，不具有幂等性，两次相同的POST请求会在服务器端创建两份资源，他们具有不同的URI。
    • 缓存问题
      • GET在浏览器中缓存。
      • POST在浏览器中不缓存

  • 进程，线程，协程

    • 含义

      • 进程

        是系统进行资源分配和调度的一个独立单位，每个进程具有自己独立内存空间。

      • 线程

        线程是进程的一个实体，是CPU调度和分派的基本单位，是比进程更小的基本单位，他通过进程获取内存空间。

      • 协程

        用户态的轻量级线程，协程的调度完全由用户控制。

    • 区别

      • 进程与线程：
        1. 进程有自己独立的内存空间，而线程没有，线程需要依赖于进程的内存空间。一个进程至少有一个线程
        2. 进程是资源调度的基本单位
        3. 同一个进程内的线程共享进程的资源，进程是资源分配的单位。
      • 线程与协程：
        1. 一个线程可以拥有多个协程，一个进程也可以拥有多个协程。
        2. 协程为异步，线程则是同步。
        3. 协程保留上一次调用状态。

  • Sleep与Wait

  • Synchronized与ReentrantLock 的区别

    • 两者都是可重入锁

      获得某一对象锁时，若自己没有释放，也可重复再次获得该对象的锁。

    • Synchronized 依赖于JVM实现，而ReentrantLock依赖于API实现(JDK层面)

    • ReentrantLock可实现公平锁，而Synchronized仅实现不公平锁。

    • 等待/通知机制 不同:

      • Synchronized 通过 notyfy() notifyAll() wait()实现等待与通知。

      • ReentrantLock 通过 Condition对象来实现：

        • 一个lock可以创建多个conditon对象，一个condition对象可以注册多个线程。

          Condition对象调用 signal() signalAll()

          ​ 唤醒已注册的线程

          Synchronized 调用 notify() || notifyAll()

          ​ 唤醒JVM选择的线程

  • ArrayList 与 LinkedList 的区别

    • 区别：ArrayList类比动态数组，LinkedList类比链表。

    • 插入元素时间复杂度

      对于ArrayList来说，主要消耗的时间在于 System.arraycopy动作。add(E) 时间复杂度为O(1),而对于add(E,index)而言，时间复杂度为O(n)。

      对于LinkedList来说，主要消耗的时间在于 for 循环找到指定的index。add(E) ** 时间复杂度为O(1),对于add(E,index)** 时间复杂度为O(n)。

    • ArrayList扩容的时机

      扩容的时机：新增元素后的minCapacity超过了当前集合的容量，则调用grow方法扩容，一般扩容到原大小的1.5倍。

    • ArrayList缩容的时机

      缩容的时机：ArrayList本身并不会自动缩容，但是提供了给外部缩容的trimToSize()方法。

    • ArrayList是否线程安全，有哪些线程安全的List

      否，Vector：给几乎所有的方法加上了synchronized关键字。所以被抛弃。

发表于 2020-05-18 | 分类于 学习 ， Java ， JVM

  • 内存区域

    Java内存结构如图所示:

    • 程序计数器

      其作用是当前线程所执行的字节码行号的指示器。类似与CPU中的PC寄存器。而对于Java的多线程，其本质上是一个处理器(也指多核处理器中的一个核)同一时间只会执行一条指令，所以对于每一个线程，都需要一个程序计数器，所以我们也称其为线程私有。如果线程正在执行的是一个Java方法，这个计数器记录的是正在执行的虚拟机字节码指令的地址，如果正在执行的是Native方法，这个计数器值则为空（Undefined）。此内存区域是唯一一个在Java虚拟机规范中没有规定任何OutOfMemoryError情况的区域。

    • 虚拟机栈

      同程序计数器，他也是线程私有的。每个方法被执行的时候都会同时创建一个栈帧(Stack Frame)用于存储局部变量表，操作栈，动态链接与方法出口等信息。每个方法被调用直至执行完成的过程，就对印着相应一个栈帧在虚拟机栈中入栈和出栈的过程。

    • 本地方法栈

      与虚拟机栈发货作用类似。区别在于，虚拟机栈为虚拟机执行Java方法(bytecode)服务，而本地方法则是为虚拟机使用到的Native方法服务。

    • 堆

      堆是JVM中管理的内存中最大的一块，是被所有线程都能够共享该内存区域。用于存放几乎所有的对象实例。

    • 方法区

      用于存放被虚拟机加载的类信息，常量，静态变量,其中也包括常量池。也被称为永久代(Permanent Generation)

    • 元空间

      对JBM规范中的方法去的实现，元空间并不在虚拟机中，而是使用本地内存。

  • Minor GC触发条件

    Eden区满时，则出发Minor GC

  • FULL GC触发条件
    Minor GC 平均晋升空间大小>老年代连续剩余空间，则触发FULL GC
    • 调用System.gc时，系统建议执行FULL GC，但是不一定执行。
    • 老年代空间不足。
    • 方法区空间不足。
    • 通过Minor GC后进入老年代的平均大小大于老年代的可用内存。
    • 由Eden区，From Space区向To Space区复制时，对象大小小于To Space，则把该对象转存老年代，且老年代的可用内存小于该对象的大小。
